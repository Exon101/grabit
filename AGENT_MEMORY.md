### Data flow

**Popup → Background → Extractor → Media results**

1. User opens popup on a supported tab
2. `popup.js` sends `{type:'scanActiveTab'}` to background
3. Background calls `runExtractorForTab(tab, settings)` which:
   - Checks `siteKeyFromUrl(url)` → returns the registry key
   - Checks `settings.sites[key]` — if false, returns `[]`
   - Calls `ext.extract(tab, {settings})` which does `fetch(url, {credentials:'include'})`
     and parses the HTML/JSON for media URLs
4. Background caches results per-tab for 30s (so popup re-opens are instant)
5. Background tries to send `{type:'scanResult', media}` to the content script
   (for the overlay to update) — failure is silently ignored (content script
   may not be loaded yet)

**Download: 3-tier fallback**

Tier 1 (`tryChromeDownload`): Direct `chrome.downloads.download({url, filename})`.
- Works for URLs that don't require Referer
- Skipped entirely if `variant.needsReferer === true`

Tier 2 (`tryFetchBlobDownload`): `fetchWithCredentials(url)` → Blob →
`URL.createObjectURL(blob)` → `chrome.downloads.download({url: objectUrl})`.
- Bypasses Referer via the DNR rules (the static rules in `dnr_rules.json`
  rewrite Referer/Origin on the network layer before the request goes out)
- Has a 60s timeout; large files may need longer

Tier 3 (last resort): `chrome.tabs.create({url, active: false})` — opens the
URL in a background tab. The browser will either display the media inline or
trigger its own download. We mark the entry as `'opened-in-tab'` and notify.

**Hover overlay mount sequence**

1. `content.js` boots at `document_idle`
2. Loads settings from `chrome.storage.sync`
3. If `settings.showHoverOverlay` is true, calls `scheduleOverlayMount()` (800ms debounce)
4. `tryMountOverlay()` checks `window.GrabItOverlay` (set by `hover-overlay.js`)
5. If not yet loaded, `ensureOverlayLoaded()` fetches `hover-overlay.css` (as
   a `<link>`) and `hover-overlay.js` (as an injected `<script>` text)
6. `window.GrabItOverlay.mount({settings, onDownload, getMedia})` builds the
   Shadow DOM root and returns a controller with `applySettings`, `updateMedia`,
   `unmount`, `showRecentToast`
7. SPA navigations (`pushState`/`replaceState`/`popstate`) trigger `onSpaNav()`
   which invalidates the scan cache and re-mounts the overlay

## Resolved bugs (with root cause + fix)

### 1. `downloads` permission is NOT required in MV3 for `chrome.downloads.download()`

**Symptom**: Earlier manifest requested `"downloads"` permission. Chrome Web Store
review flagged it as overly broad.

**Root cause**: Confusion with MV2 where `downloads` was indeed needed. In MV3,
`chrome.downloads.download()` works with just the host permission for the URL
being downloaded. Our host_permissions already cover all the supported sites.

**Fix**: Removed `"downloads"` from `permissions` array in manifest. Verify by
loading unpacked — `chrome.downloads.download()` still works.

### 2. YouTube signature bridge needs MAIN world, not isolated

**Symptom**: Bridge script (`lib/yt-bridge.js`) couldn't see `ytInitialPlayerResponse`
on `window`.

**Root cause**: Content scripts run in an "isolated world" — they see a separate
`window` from the page. YouTube's `ytInitialPlayerResponse` is set by a page script,
so it's on the page's `window`, not the content script's.

**Fix**: Inject `yt-bridge.js` as a `<script>` element with `textContent = code`
(rather than as a `chrome.scripting.executeScript` content-script injection). The
script then runs in the page's MAIN world. The bridge is listed in
`web_accessible_resources` so the content script can `fetch()` its URL.

### 3. Content scripts can't easily be ES modules

**Symptom**: Tried `import { foo } from '../lib/utils.js'` in `content/content.js`.
Chrome silently failed to load the script.

**Root cause**: MV3 content scripts registered via the manifest `content_scripts`
key are loaded as classic scripts, not modules. You can register modules dynamically
via `chrome.scripting.executeScript({world:'MAIN', files:[...]})` but the
declarative manifest approach doesn't support modules.

**Fix**: Keep `content.js` as a plain IIFE — no imports. It uses `chrome.runtime`
and `chrome.storage` directly. If it needs shared helpers, it inlines them or
sends a message to the background (which can use modules).

### 4. DNR rules can't modify Referer for `xmlhttprequest` in some Chrome versions

**Symptom**: DNR rule with `resourceTypes:['xmlhttprequest']` to set Referer
on a fetch() from the service worker didn't apply.

**Root cause**: Service-worker fetches are typed as `xmlhttprequest` in MV3, but
Chrome <116 had a bug where DNR `modifyHeaders` didn't fire on SW-initiated
requests in some edge cases. Also, `referer` is a "forbidden" header that DNR
*can* modify but only since Chrome 116+.

**Fix**: Set `minimum_chrome_version` to `"116"` in manifest. Also added
`'other'` to the `resourceTypes` array in DNR rules as a catch-all. The
fetch fallback in `lib/restriction-bypass.js` includes `Sec-Fetch-*` headers
which some CDNs check instead of Referer.

### 5. Inline scripts in popup/options break CSP

**Symptom**: Tried to inline a small JS snippet in `popup.html`. Chrome
blocked it: "Refused to execute inline script because of CSP".

**Root cause**: MV3 `extension_pages` CSP defaults to `script-src 'self'`.
No inline scripts allowed.

**Fix**: All JS in external files. The one place we *do* inject inline JS
is the YouTube bridge — that goes into the *page* world via a `<script>`
tag, which is allowed because content scripts (not extension pages) inject
it. The `web_accessible_resources` entry for `lib/yt-bridge.js` is required
so the content script can fetch it as text and inject it.

### 6. SPA navigations left stale overlay and stale scan results

**Symptom**: On YouTube, clicking a related video updated the page but the
GrabIt overlay still showed the old video's variants.

**Root cause**: YouTube is a SPA — `pushState` changes the URL without a
full page reload, so `content.js` doesn't re-run, and the background's
per-tab scan cache stays stale.

**Fix**: Patch `history.pushState` and `history.replaceState` in
`content.js` to fire a custom `onSpaNav` callback. Also listen for
`popstate`. On SPA nav: (1) invalidate the background scan cache via
`chrome.tabs.onUpdated` listener (background side), (2) re-mount the
overlay (content side). The 800ms debounce prevents thrashing on rapid
SPA transitions.

## Permissions rationale

| Permission | Why it's needed | Why we don't avoid it |
|------------|-----------------|----------------------|
| `activeTab` | Read current tab URL to dispatch extractor | Required — no workaround |
| `scripting` | Inject yt-bridge.js into MAIN world | Could use manifest content_scripts, but MAIN world requires scripting API in MV3 |
| `storage` | Persist settings (sync) + recent downloads (local) | Required for any persistence |
| `tabs` | Read tab title + favicon for popup display | Could use activeTab only, but tabs gives us URL change events for SPA detection |
| `contextMenus` | Right-click "GrabIt" entry | UX nicety; could be dropped |
| `notifications` | Download complete/error toasts | UX; could be dropped |
| `declarativeNetRequest` | Spoof Referer/Origin for CDN-hosted media | Required — many CDNs (Twitter, Reddit, TikTok, Twitch) 403 without correct Referer |
| `declarativeNetRequestFeedback` | (Currently unused) Future: observe rule matches for debugging | Could be dropped |
| Host perms (10 sites) | Fetch site HTML/JSON for extraction | Each site is independently toggleable in settings |
| `downloads` | **NOT requested** — see bug #1 above | N/A |

## Known limitations (for future work)

1. **YouTube ciphered streams** — The yt-bridge handles `ytInitialPlayerResponse`
   URLs that are already unciphered, but doesn't actively decode `signatureCipher`
   parameters. For fully-ciphered videos, we return an error marker.
   **Fix**: Implement the signature decoder by fetching the player JS and
   extracting the `rsig` function. yt-dlp does this; we could port a minimal version.

2. **Bilibili DASH muxing** — Video and audio are separate variants; downloading
   just video yields a silent file.
   **Fix**: Use ffmpeg.wasm to mux the two streams after both download. ~30MB add.

3. **TikTok URL expiry** — Signed URLs expire after ~1 hour. If the user opens
   the popup but waits too long before clicking, the download fails.
   **Fix**: Re-fetch on download attempt; show "URL expired, refreshing…" state.

4. **Instagram stories** — Different URL pattern; not currently matched.
   **Fix**: Add `/stories/<user>/<id>` to the regex; story JSON shape is similar to reels.

5. **Twitch live streams** — Live streams use a different m3u8 shape (segment
   rolling window); not currently supported.
   **Fix**: Add a 9th extractor for `twitch.tv/<channel>` that grabs the live
   m3u8 and lets the user record N seconds.

6. **No i18n** — All UI text is English. Should add `_locales/` with at least
   en, zh-CN, es, fr, de.

7. **No tests** — Manual smoke testing only. Should add Playwright + extension
   testing via `playwright-chromium` with `--load-extension`.

8. **No build step** — Source is shipped as-is. Could add a Rollup/esbuild step
   to bundle modules + minify for smaller package size.

9. **Per-tab scan cache invalidation** — Currently 30s TTL + invalidation on
   `tabs.onUpdated` complete. Doesn't invalidate on user interaction (e.g.
   clicking a different YouTube video mid-popup-open). Could use a content
   script heartbeat.

## Developer notes

### ES module usage
- `background/background.js` is `"type": "module"` per manifest
- `lib/*.js` and `extractors/*.js` are ESM with `export`/`import`
- `content/content.js` and `content/hover-overlay.js` are **NOT** modules
  (they're loaded as classic scripts by the content-script mechanism)
- `popup/popup.js` and `options/options.js` are **NOT** modules either
  (loaded via `<script src>`)

### Extractor registration order
Order matters only for `listExtractors()` which is used by the popup's empty
state and the options page site toggles. The current order is alphabetical
by site name for predictability. New extractors should be added in
`extractors/registry.js` at the bottom of the array.

### URL deobfuscation
`lib/utils.js` exports `deobfuscateUrl(url)` which handles:
- `youtu.be/<id>` → `youtube.com/watch?v=<id>` (so extractor matches)
- Tracking redirect params (`?url=`, `?u=`, `?redirect=`, etc.)
- Base64-encoded wrappers (rare, but used by some aggregators)
- Known shorteners (t.co, bit.ly, etc.) are passed through; the background
  uses `fetch(url, {redirect:'follow'})` to resolve them via HEAD probe

### DNR rule IDs
- Static rules in `background/dnr_rules.json`: IDs 1–10
- Dynamic rules from `buildStaticDnrRules(100)`: IDs 100–999
- Per-session dynamic rules (not yet used): IDs 5000–5999
- Don't reuse IDs across static and dynamic — Chrome will reject the rule

### Settings sync points
- `chrome.storage.sync` key: `'settings'`
- Stored as `{ v: settingsObject, e: 0 }` (TTL wrapper for consistency with
  `storageGet`/`storageSet` helpers, though settings have no TTL)
- Content script listens for `chrome.storage.onChanged` in sync area and
  applies settings to the overlay in real-time
- Options page debounces saves by 300ms to avoid spamming on rapid input

### Log levels
- `logger.debug(...)` — verbose; only shown when `settings.debugLogging` is true
- `logger.info(...)` — default; lifecycle events (boot, scan results, downloads)
- `logger.warn(...)` — recoverable failures (extractor returned no media, DNR
  rule failed to apply)
- `logger.error(...)` — unrecoverable errors (extractor threw, IPC handler failed)

Set `settings.debugLogging = true` in the options page → Advanced → Enable
debug logging, then inspect the service worker via `chrome://extensions` →
"Inspect views: service worker".

### Branding
- Logo: indigo gradient rounded square + white download arrow
- Icon SVG reused across popup, options, and hover overlay for consistency
- Accent color is configurable; default `#6366f1` (indigo-500)
- Font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', 'Inter', Roboto, ...`
  — no web font loaded, uses system default for instant load

### CI / Release strategy (`.github/workflows/build.yml`)
- **Trigger**: every push to `main` (and `workflow_dispatch`)
- **Rolling release** tagged `latest`: a single prerelease that holds exactly
  one asset, `grabit-extension.zip`. On every commit, the asset is replaced
  via `gh release upload --clobber` — the old zip is effectively deleted.
- **Archive releases** tagged `v{N}.x-archive`: created once, when the manifest
  major version bumps from N to N+1. Before replacing `latest`, the workflow
  downloads the current `latest` zip and uploads it to a new permanent
  release as `grabit-extension-v{full-version}.zip`. Archive releases are
  write-once — the workflow checks `gh release view` first and refuses to
  overwrite an existing archive.
- **Major version detection**: read `manifest.json` from `HEAD` and `HEAD~1`,
  compare the first dot-segment of `.version`. Only triggers when `prev < current`
  (strictly increasing) — downgrades don't archive.
- **Concurrency**: `cancel-in-progress: false` to prevent losing an archive
  mid-flight if a new push arrives.
- **Zip exclusion**: `.git/`, `.github/`, `*.zip`, `.gitignore`, `*.DS_Store`
  are stripped from the published zip so it contains only runnable extension
  files (manifest.json at the root).
